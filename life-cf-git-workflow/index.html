How a model change gets to production

A release-train walkthrough for actuaries — from feature branch on your laptop to tagged deploy on Oracle Cloud. Mix of approved target workflow and what exists in the repo today.

Dashed pipeline boxes = target spec (deploy-preprod.yml / deploy-prod.yml not committed yet). Solid boxes = parsed from ci.yml.

Actuarial mapping Model change control in software looks like git on main — one integration branch that should always be deployable in principle. Nothing goes to prod from a laptop folder copy.

Dev · local You branch feature/assumption-schema from main. This is your working paper — fast iteration with just test on your machine.

Commits stay on the feature branch until review. Preprod and prod are shared environments — you don't point them at half-baked work.

Implemented today Open a PR → GitHub Actions runs ci.yml: checkout, install uv/just, sync dependencies.

Implemented today CI must pass lint, unit/integration tests, and a summary regression before merge. This is the gatekeeper — like an automated model check.

Merge the PR to main. Main still doesn't auto-deploy to prod — it only receives integrated, reviewed code.

Target workflow When it's release time, cut release/2026-Q3 from main. Release branches are where you stabilise a quarterly train.

Target spec Push release/* → deploy-preprod (planned workflow) SSHs to the Oracle VM and runs docker compose -p life-preprod up.

On preprod you run the full regression (REGRESSION_MODE=both) — the software equivalent of parallel UAT. Sign-off is manual for now.

RC fixes land on the release branch, not directly on prod. Environments are cattle — rebuild preprod from git, don't hand-tweak the server.

Merge release to main and tag v2026.3.0. Prod runs an immutable tagged version — you never git pull on prod and hope.

Target spec Tag triggers deploy-prod with a GitHub Environment approval gate, then the life-prod compose stack on the same VM — separate DB, separate OCI bucket.
